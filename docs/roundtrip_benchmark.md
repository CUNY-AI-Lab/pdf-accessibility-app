# Gold-To-Stripped Round-Trip Benchmark

Updated: 2026-04-06

This document defines a benchmark shape for stronger verification than:

- `veraPDF` alone
- current deterministic fidelity checks alone
- corpus pass rates alone

The core idea is:

1. start from a known-good accessible PDF
2. strip benchmark-target accessibility semantics while preserving visible content
3. run the stripped PDF through the remediation pipeline
4. compare the remediated output back to the gold PDF

This does not prove universal accessibility. It does create a much stronger verification loop for the subset of accessibility that should be recoverable from the document itself.

See also [roundtrip_gap_analysis.md](./roundtrip_gap_analysis.md) for the first application-level defect classes surfaced by the seeded corpus.
See also [gold_qualification_checklist.md](./gold_qualification_checklist.md) for the stricter promotion rubric used for external candidates.

Current seeded golds now include:

- a prose/manual document (`bmuv_manual`)
- an official form (`uscis_i9`)
- a long-form research-style technical report (`fallmt2022`)
- a short academic paper / abstract (`pdfa_academic_abstract`)
- a long-form book / chapter document (`pdfa_book_chapter`)

## Gold requirements

A round-trip gold PDF must be stronger than "something that looks tagged".

Minimum bar:

- independently produced, not generated by this app
- passes `veraPDF` with explicit `ua1` flavour
- has a believable accessibility provenance
  - publisher-provided PDF/UA
  - institutionally remediated accessible original
  - manually curated accessible document with known source
- survives a short human spot-check for practical accessibility
  - title is announced correctly
  - headings and reading order are sane
  - links and bookmarks are usable when present
  - tables and forms are navigable when present

Documents that fail any of those checks should not be promoted into the round-trip gold corpus.

In particular:

- app-generated tagged outputs are not golds
- local probe outputs are not golds
- compliance alone is necessary but not sufficient
- documents with uncertain provenance should be kept as candidates, not golds

## Why this benchmark exists

The current release gate already tests:

1. PDF/UA compliance
2. structural/content fidelity
3. completion without unresolved blocking remediation

That is good, but it still leaves an important question open:

Can the system reconstruct an accessible reading experience that is close to a known good accessible document, rather than merely producing a technically compliant output?

The round-trip benchmark is meant to answer that.

## What the benchmark can verify well

This approach is strong for accessibility features that are recoverable from visible or near-visible evidence:

- reading order
- heading hierarchy
- list semantics
- many table structures and header relationships
- visible form labels
- link labeling when the visible label is present
- page and document language when visible evidence is strong
- assistive transcript quality for ordinary text documents

## What it cannot fully verify

This benchmark should not be treated as proof that every accessibility feature is automatically recoverable.

It is weak or incomplete for:

- author-written alt text whose meaning is not visible in the image
- hidden `/ActualText` repairs for glyph issues with no visible clue
- metadata-only title or language signals
- multilingual span semantics that do not appear clearly in visible content
- complex tables whose intended relationships exceed visible geometry
- cases where the gold PDF encodes author intent that a stripped PDF no longer exposes

Those belong in a separate hidden-semantics lane or in human review.

## Benchmark lanes

### 1. Recoverable-Core Lane

This should become the main automatic round-trip lane.

It strips:

- `/StructTreeRoot`
- `/MarkInfo`
- `/Lang`
- `/Metadata`
- bookmarks/outlines
- page `/Tabs`
- `/StructParent` and `/StructParents`
- `/TU`
- marked-content `/MCID`
- marked-content `/ActualText`
- marked-content `/Alt`
- marked-content `/Lang`
- marked-content `/E`
- link/widget annotation `/Contents`

It keeps:

- visible page content
- page geometry
- ordinary annotations/widgets as visible objects
- fonts, images, and drawing commands

Current implementation:

- [backend/scripts/strip_accessibility.py](../backend/scripts/strip_accessibility.py)

### 2. Hidden-Semantics Lane

This is not a pure automatic target yet.

Use this lane for evaluating:

- author-authored alt text
- metadata-only titles
- invisible language overrides
- special `/ActualText` repairs

These should be benchmarked with targeted assertions, not by assuming round-trip recoverability.

## Comparison targets

The remediated PDF should be compared against the gold PDF on three layers.

### Structural comparison

- headings restored
- lists restored
- tables restored
- forms labeled
- bookmarks restored when expected

### Assistive transcript comparison

Compare what assistive technology would effectively encounter, not just raw PDF text extraction.

At minimum, compare:

- reading order
- heading announcements
- table navigation text
- form field presence, role/type, and accessible name adequacy
- link label text

### Meaning-preservation assertions

Each benchmark file should include a small set of must-preserve assertions such as:

- dates
- amounts
- negation
- checkbox/radio state
- table header-to-cell meaning
- figure function
- explicit warnings/instructions

These should be hand-written and small in number.

## Corpus annotation model

Each gold PDF should eventually carry a sidecar manifest with:

- `id`
- `document_family`
- `product_priority`
- `workflow_tags`
- `gold_pdf`
- `stripped_pdf`
- `lane`
- `recoverable_assertions`
- `hidden_semantics_assertions`
- `task_checks`

Recommended classification for every assertion:

- `recoverable_from_visible_evidence`
- `requires_hidden_author_semantics`

Only the first class should be used as a hard automatic pass requirement in the round-trip lane.

Minimal manifest example:

```json
{
  "document_family": "technical_report",
  "product_priority": "primary",
  "workflow_tags": ["toc", "bookmarks", "front_matter", "tables"],
  "recoverable_assertions": [
    {"id": "lang", "kind": "document_lang", "expected": "en-US"},
    {"id": "title", "kind": "title_contains", "expected": "Annual Report"},
    {"id": "flow", "kind": "ordered_text", "expected": ["Executive Summary", "Revenue grew"]},
    {"id": "field_name", "kind": "field_accessible_name", "field_name": "name", "expected": "Name"},
    {
      "id": "guide_link",
      "kind": "link_contents",
      "uri": "https://example.com/guide",
      "expected": "Open the guide"
    }
  ],
  "hidden_semantics_assertions": [
    {"id": "chart_alt", "kind": "text_contains", "expected": "Sales increased 12 percent"}
  ]
}
```

## Universal Workflow Matrix

This app is a universal remediation workflow, so the corpus report should be read on two axes:

- family coverage
  - manual, technical report, conference paper, form, and later journal article / table-heavy report
- universal invariants
  - page count preserved
  - raw text preserved
  - reading order preserved
  - document language restored
  - title restored

Family-specific focus areas still matter, but they should sit on top of those cross-family invariants rather than replace them.

## Stripping command

Create a stripped benchmark input from a known-good accessible PDF:

```bash
cd backend
PYTHONPATH=. uv run python scripts/strip_accessibility.py \
  --input /path/to/gold-accessible.pdf \
  --output data/benchmarks/roundtrip/mydoc_stripped.pdf
```

Optional summary artifact:

```bash
cd backend
PYTHONPATH=. uv run python scripts/strip_accessibility.py \
  --input /path/to/gold-accessible.pdf \
  --output data/benchmarks/roundtrip/mydoc_stripped.pdf \
  --summary-json data/benchmarks/roundtrip/mydoc_strip_summary.json
```

Parallel corpus sweep:

```bash
cd backend
PYTHONPATH=. uv run python scripts/roundtrip_corpus_benchmark.py \
  --workflow-profile assistive-core \
  --jobs 4
```

`--jobs` parallelizes manifests at the case level using isolated per-case workflow databases. The runner now defaults to `4`; reduce it if shared services like `docling-serve`, local PDF tagging, or remote model endpoints start to saturate.

## Comparison command

Compare a remediated candidate back to the gold PDF:

```bash
cd backend
PYTHONPATH=. uv run python scripts/roundtrip_compare.py \
  --gold /path/to/gold-accessible.pdf \
  --candidate /path/to/remediated-output.pdf \
  --manifest /path/to/mydoc.roundtrip.json \
  --json-out data/benchmarks/roundtrip/mydoc_compare.json \
  --markdown-out data/benchmarks/roundtrip/mydoc_compare.md
```

The comparison report currently scores:

- raw extracted text similarity
- structure-transcript similarity and reading-order recovery
- document language/title restoration
- form field presence and role/type recovery
- bookmark restoration, split into recoverable visible-evidence titles versus higher-order grouping/front-matter semantics
- widget accessible-name restoration
- descriptive link text restoration
- recoverable and hidden-semantics assertions from a sidecar manifest

For forms, exact `/TU` replay is often too strict to be the only signal. The comparison layer now separates:

- field presence recovery
- field role/type recovery
- exact accessible-name replay
- manifest-driven accessible-name adequacy checks

That lets the benchmark distinguish true assistive regressions from harmless wording differences when the name/role/value experience is still correct.

Recoverable form assertions can now be expressed in a few stronger ways:

- `field_present`
- `field_type`
- `field_accessible_name` with exact `expected`
- `field_accessible_name` with `expected_any`
- `field_accessible_name` with `required_terms_all` and optional `required_terms_any`

This is the preferred way to encode assistive expectations such as:

- the field exists
- the field is still the right control type
- the accessible name preserves the critical content words
- the accessible name still carries the needed disambiguating context

## Corpus runner

Run the round-trip corpus lane over every `*.roundtrip.json` manifest under
`backend/benchmark_manifests/roundtrip_corpus`:

```bash
cd backend
PYTHONPATH=. uv run python scripts/roundtrip_corpus_benchmark.py
```

This runner defaults to the `assistive-core` remediation profile, which keeps the full downstream remediation and verification loop in place while skipping only figure alt-text generation:

- TOC structure assistance
- grounded text repair
- table header intelligence
- widget rationalization
- form accessible-name intelligence
- validation and fidelity/review surfaces

Use the full workflow only when you specifically need figure/alt-text behavior:

```bash
cd backend
PYTHONPATH=. uv run python scripts/roundtrip_corpus_benchmark.py --workflow-profile full
```

The seeded corpus currently includes:

- `bmuv_manual.roundtrip.json`
- `uscis_i9.roundtrip.json`

Each run writes:

- per-case stripped inputs and round-trip reports
- a corpus summary CSV
- a corpus summary JSON
- a corpus report Markdown file

## Immediate next step

The next useful implementation step is a very small gold set:

1. one independently accessible academic article or research-style report
2. one independently accessible journal issue or multi-article academic PDF
3. one table-heavy academic or technical PDF
4. one manual/admin prose PDF
5. one form PDF as a secondary stress suite

Each should have:

- a gold accessible PDF
- a stripped derivative
- a short assertion file
- a comparison report against the remediated output

That will tell us quickly whether the benchmark is actually discriminating the problems we care about:

- readability
- meaning preservation
- practical accessibility

Forms matter, but they are not the primary audience baseline for this app. The round-trip corpus should therefore expand academic and research-style documents first, then keep forms as a separate stress lane.
